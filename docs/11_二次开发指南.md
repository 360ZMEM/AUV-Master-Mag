## 1. 必须先理解的硬约束（改之前请读）

任何二次开发都不得破坏以下不变量，否则会引入难以察觉的回归：

| 约束 | 内容 | 守门点 |
|---|---|---|
| **C1 磁横偏隔离** | 磁横偏（`magnetic_cross_track_offset_m`）**绝不**喂入主 PCA 中心线拟合器 | LOCK→TRACK 闸门用 `fit_result.covariance_xy_m2` 较小特征值代理 EKF `P_yy`；横偏只在 TRACK 段做压线修正 |
| **C5 直缆零回归** | case1–4 直缆/转弯/噪声/姿态指标不得回退 | 改动后必须跑 `tools/visualize.py --progress` 对照 `viz/baseline.py` |
| **GUI/Logic 分离** | `viz/figures.py` 是**唯一** import matplotlib 的模块 | 其余 `viz/*` 与全部核心模块保持纯数据/纯逻辑 |
| **单一感知契约** | 控制层与任务层只读 `PerceptionState`，不直接碰传感器 | [`perception/state.py`](../src/auv_mag_tracking/perception/state.py) |
| **指标单一事实来源** | 实时仪表盘与离线报告共用 `viz/metrics.py` 同一函数 | 不允许在别处另写一套指标 |

> 经验教训：曾尝试"独立磁法曲率中心线估计器"，因物理信噪比被证伪（见 [`09`](09_实施进度与成果.md) §4）。
> 新增估计器前，先用逐帧诊断确认信号强度高于噪声基底，否则属不可执行域。

```

传感器        sensor_model.py       Magnetometer(HF) / Sonar / IMU / BurialDepthObserver

任务决策      mission_manager.py    三态 FSM（SEARCH / LOCK / TRACK + EMERGENCY）
```

每层都是"配置驱动 + 单一职责"，扩展时优先在既有层内加字段/加方法，而非新增并行通路。

## 3. 新增一个 case（最常见任务）

场景是纯配置，不用改任何逻辑。步骤：

1. 打开 [`config/__init__.py`](../src/auv_mag_tracking/config/__init__.py) 的 `build_default_scenarios()`。
2. 以最接近的 case 为模板深拷贝，覆盖差异字段（信号/传感器/声呐/环境/跟踪）。
3. 路由几何：直线用 `cable_route_mode="linear"`；转弯用 `"spline"` + 多控制点；缓弯用 `"sine"` + 波长/振幅。
4. **曲率守门**：保持 `validate_curvature_on_build=True`，构建期会校验 `min_cable_curvature_radius_m`，
   防止生成不可跟踪几何（波长越长→曲率越缓→可设更大 minR，见 [`06`](06_场景配置指南.md) §2.5）。
5. 验证：`python3 tools/visualize.py --case <新case>`，看 `results/<时间戳>/<新case>/report.md`。

> 若希望新 case 进入 `--all` 默认集合，需在 [`tools/visualize.py`](../tools/visualize.py) 的 `DEFAULT_CASES` 加名字
>（当前默认仅 case1–5；case6/8/高保真需 `--case` 单跑）。

各 case 的字段含义与实测指标见 [`06_场景配置指南.md`](06_场景配置指南.md) §3。

## 4. 扩展感知算法

### 4.1 在 PerceptionState 加一个输出量

[`perception/state.py`](../src/auv_mag_tracking/perception/state.py) 的 `PerceptionState` 是唯一感知契约。

1. 在 `PerceptionState` 加字段（给默认值，避免破坏既有构造点）。
2. 在 [`perception/orchestrator.py`](../src/auv_mag_tracking/perception/orchestrator.py) 的 `update()` 里计算并填入。
3. 若控制器要消费，在 [`controller.py`](../src/auv_mag_tracking/controller.py) 读取该字段（**只读**）。
4. 若要可视化/落盘，去 §6 给 recorder 加通道。

### 4.2 新增一个子估计器（filters/peaks/fitter 同款模式）

`perception/` 包内每个子模块都是"配置进、状态留、纯函数出"的小类：
- 构造时只吃 config；
- `update(...)` 接收当帧输入、返回结果，不做 I/O、不碰 matplotlib；
- 在 `orchestrator.update()` 里实例化并串入。

**红线**：任何新中心线估计量都不得进入 `fit_result.covariance_xy_m2` 闸门（守 C1）。
横偏类信号只能作为 TRACK 段压线修正，参考既有 `cross_track.py` 的接法。

### 4.3 调阈值而非加分支

FSM 转移判据集中在 [`mission_manager.py`](../src/auv_mag_tracking/mission_manager.py) 的 `MissionThresholds`
（如 `mag_lock`/`release`、`signal_hold_s`、`cov_perp_converged`）。优先调阈值，
而不是在 FSM 里加新状态——三态对应三个物理阶段，新增态会破坏可去抖性。

## 5. 扩展控制律

[`controller.py`](../src/auv_mag_tracking/controller.py) 的 `ZigZagController.update(pose, perception)`
输出 `GuidanceCommand`（`desired_heading_deg` / `speed_mps` / `mode` / `yaw_rate_deg_s` / ...）。

设计原则（已写入 project memory，务必遵守）：
- **纵向基准与横向压线分轴**：沿缆基准航向走一条低通；横偏修正单独加在其上。
- **磁横偏符号一致性**：`magnetic_cross_track_offset_m` 在不同 case 间符号/尺度尚未统一标定
 （见 [`09`](09_实施进度与成果.md) §7 已知边界）——这是当前最优先的待标定项。改控制律前先确认符号约定。

## 7. 测试与回归流程（合并前必跑）

```bash
# 1) 全套件单测（60 用例，含 C1/C5 守恒断言）
python3 -m unittest discover -s tests

# 2) 全 case 实测矩阵（看指标是否符合预期）
python3 tools/visualize.py --all

# 3) before/after 进度对照（对照 viz/baseline.py 固化基线，自动给"改善/持平/回退"）
python3 tools/visualize.py --progress
```

判定标准：
- 单测全绿；
- `--progress` 中 case1–4 不出现"回退"（C5 零回归）；
- 目标 case 指标按预期改善，且未把其它 case 推过 15° 硬约束。

> 环境提示：`python3` 指向 conda（TraeAI-3）；`results/` 已被 `.gitignore` 忽略（图像不进版本库，
> 故进度基线固化在 `viz/baseline.py` 代码常量里，保证干净 checkout 可复现）。
> 测试输出里的 curvature 构建告警是预期的几何校验信息，过滤即可。

## 8. 变更管理约定

- **分支**：重大重构/算法改动先在独立分支验证通过（§7 三连），再合并 `main`；推送远程需手动确认。
- **commit message**：用 `Write` 写入临时文件再 `git commit -F`（避免 shell 转义问题），提交后删除该文件。
- **暂存**：按文件名显式 `git add`，禁用 `git add -A`（防误纳 `results/`、临时文件、密钥）。
- **最小改动**：bug 修复不顺手重构；一次性操作不抽象出 helper；不为假想需求预留扩展。
