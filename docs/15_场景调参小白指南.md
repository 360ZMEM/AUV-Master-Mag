# 场景调参小白指南

本文给第一次调本项目的人一个直接入口：先判断自己遇到的是哪类场景，再只动少数关键参数。不要一上来网格搜参；本项目的闭环里，很多参数互相耦合，盲扫很容易把“投影进度变高”误判成“真的到终点”。

## 0. 先看三个结果，不只看 route

每次跑完 `tools/visualize.py`，至少同时看：

| 指标 | 怎么理解 |
| --- | --- |
| `stop_reason` | `endpoint` 才是真正到终点；`duration` 只是时间用完。 |
| `route completion` | 沿路线投影的进度，可能出现假阳性。 |
| endpoint 距离 / `final_route_distance_m` | 终点距离和横向距离都小，才说明路线真的闭合。 |
| `track_mean_cross_track_m` | TRACK 段是否真的贴线。 |
| `track_mean_vehicle_heading_error_deg` | 控制闭环是否沿着电缆切向走。 |

建议命令：

```bash
/Users/bytedance/miniconda3/bin/python tools/visualize.py --case case6
/Users/bytedance/miniconda3/bin/python tools/visualize.py --maze
/Users/bytedance/miniconda3/bin/python tools/visualize.py --all
```

`--all` 会输出 `case1`–`case6`；`--maze` 会输出 `case_maze_sonar`、`case_maze_sonar_dropout` 和 `case_maze_no_sonar`。如果只看 `--all` 的 showcase，不要把 `case6` 的 `mean heading error` 直接等同于“车辆贴线失败”，还要打开单场景 report 看 TRACK 段横向误差、TRACK 段车辆航向误差、最终横向距离和 route 进度。

## 1. 直线或轻微弯曲：先不要动大结构

适用场景：`case1`–`case4`，以及大多数直缆/缓弯场景。

优先调：

| 问题 | 优先参数 | 调参方向 |
| --- | --- | --- |
| 磁峰太少 | `min_peak_strength_nt` | 适当降低，但不要低到噪声也触发。 |
| 峰值抖动 | `median_window_samples`、`hysteresis_fraction` | 增大窗口或迟滞。 |
| 进入 TRACK 太早 | `fsm_cov_perp_converged_m2`、`fsm_yaw_err_converged_deg` | 收紧门槛。 |
| TRACK 后横向偏大 | `track_active_zigzag_angle_deg` | 小幅调整，通常不要大改。 |

不建议优先动：

- `reacquire_region_*`：这是 maze/失锁重捕获用的，直缆场景默认不该依赖它。
- 大幅提高 zig-zag 角度：容易让稳定跟踪变成反复横切。

## 2. 强弯融合压力：先接受 case6 是表达力边界

适用场景：`case6`，连续 S 弯、声呐和磁感知融合。

当前稳定基线：

| 场景 | 当前指标 | 结论 |
| --- | --- | --- |
| `case6` | mean heading error 约 `27.6deg` | 这是 fused-heading 对最近电缆切向的全程均值口径；AUV 在 TRACK 段可贴线，但感知航向表达仍未收口。 |

当前诊断口径：

| 指标 | 典型值 | 怎么读 |
| --- | --- | --- |
| 全程 fused-heading mean error | `27.6deg` | 说明连续 S 弯下航向估计滞后/偏差明显。 |
| TRACK 段平均横向误差 | 约 `2.8m` | 说明车辆进入 TRACK 后并不是完全跟丢。 |
| 默认 200s route progress | 约 `69%` | 主要受早期 SEARCH/ALIGN 和仿真时长影响。 |
| 延长并启用 endpoint 验收 | 可到 endpoint 附近 | 说明 case6 不是纯控制无法跟踪，而是“感知航向指标未收口 + 默认时长短”的混合问题。 |

优先调：

| 问题 | 优先参数 | 调参方向 |
| --- | --- | --- |
| 局部切向滞后 | `local_path_local_line_window` | 小窗口更灵敏，大窗口更稳。当前 maze/case6 相关实验常用 `5`。 |
| 航向观测太噪 | `local_path_heading_blend` | 适当增大可更信任声呐/局部航向，但过大会跟噪声走。 |
| 局部路径过早接管 | `local_path_min_confidence`、`local_path_max_residual_m` | 收紧门控。 |

不要用来“硬救”case6：

- 单纯把 zig-zag 角度调大。
- 把局部状态机直接接入任务 FSM 锁存。
- 只追求 route progress，不看 mean heading error。

## 3. maze 有声呐：调重捕获区域，不调大范围搜索

适用场景：`case_maze_sonar`。

当前稳定基线：

| 场景 | 当前指标 | 结论 |
| --- | --- | --- |
| `case_maze_sonar` | `route=99.5%`、`stop=endpoint` | 1x maze 有声呐可闭环。 |
| `case_maze_sonar_dropout` | `mean_err=3.8deg`、`route=1.5%`、`stop=duration` | 局部航向看起来好，但全局任务没有前进；maze 当前依赖持续声呐补观测。 |

关键参数：

| 参数 | 作用 | 增大后通常会怎样 |
| --- | --- | --- |
| `reacquire_region_forward_distance_m` | 在最后可信点前方放候选区域 | 更敢向前找，但可能越过正确转弯入口。 |
| `reacquire_region_turn_lateral_offset_m` | 在转弯侧放候选区域 | 更容易找 U-turn 后的 lane，但过大可能跑偏。 |
| `reacquire_region_half_width_m` | 区域横向宽度 | 更宽更容易覆盖目标，也更容易选错。 |
| `reacquire_region_half_length_m` | 区域纵向长度 | 更长更容错，但控制更钝。 |
| `reacquire_region_control_enabled` | 是否让任务 FSM 进入区域重捕获控制态 | 有声呐 maze 可开；普通场景不要默认依赖。 |
| `reacquire_region_max_duration_s` | 单次区域重捕获最长持续时间 | 不一定越大越好；当前 maze 需要频繁回 TRACK/LOCK 补观测。 |

当前代表点结论：

- `forward=48m, lateral=60m, half_width=24m, half_length=36m` 是当前有效点。
- 更保守或更激进都退化。
- `max_duration=75s/120s` 都退化，说明不是简单“在重捕获态待久一点”就能更好。

## 4. maze 无声呐：先当作可观测性边界

适用场景：`case_maze_no_sonar`。

当前稳定基线：

| 场景 | 当前指标 | 结论 |
| --- | --- | --- |
| `case_maze_no_sonar` | route 约 `0.0%` | 仅磁感知缺少足够的纵向切向观测，当前不设强通过目标。 |

可以诊断：

- 磁峰是否能触发：看 `peak_detected` 和 `tracking_strength_nt`。
- 是否有足够连续的 cross-track 信息：看 `magnetic_cross_track_quality`。
- 是否只是初始捕获失败，还是捕获后立刻丢失。

不建议：

- 直接把有声呐 maze 的 `reacquire_region_control_enabled=True` 套到无声呐。
- 靠无限 zig-zag 搜索刷 route progress。

## 5. 噪声变大：先稳传感，再动控制

适用场景：声呐误差变大、磁噪声变大、姿态扰动变强。

优先顺序：

1. 先调传感/滤波：`median_window_samples`、`hysteresis_fraction`、`noise_floor_time_constant_s`。
2. 再调峰值门槛：`min_peak_strength_nt`、`peak_cooldown_s`。
3. 再调局部路径门控：`local_path_min_confidence`、`local_path_max_residual_m`。
4. 最后才调控制：zig-zag 角、区域尺寸、状态机 streak。

## 6. 时间不够：先判断是不是“快到了”

如果 `route` 很高但 `stop=duration`，不要马上改算法。先看 endpoint 距离：

- endpoint 距离已经很小：可以适当增加 `duration_s`。
- endpoint 距离很大：说明 route progress 可能是假阳性，应该回到感知/区域选择诊断。

本次 maze 经验：

- 1x `case_maze_sonar` 在 `2200s` 时 route 已经到约 `85%`，但没有 endpoint。
- 增加到 `2600s` 后达到 `stop=endpoint`。
- 这说明该场景主要是时间预算不足，不是区域方向完全错误。

## 7. 最小调参流程

推荐每次只做这 5 步：

1. 记录当前命令和指标。
2. 只改一个参数或一个参数组。
3. 跑目标场景。
4. 跑 `case1`–`case5` 回归，确认主线不坏。
5. 把有效点写进文档；把失败点也写清楚，避免重复尝试。

当前主线回归要求：

| 回归项 | 要求 |
| --- | --- |
| `case1`–`case5` | 平均航向误差保持 `< 15deg`，且不明显劣化。 |
| `case6` | 保持约 `27.6deg`，不因 maze 改动进一步变差。 |
| `case_maze_sonar` | 1x 下达到 `stop=endpoint`。 |
| `case_maze_no_sonar` | 允许失败，但必须记录为可观测性边界。 |
