## 1. 估计器现状

### 1.1 估计器在系统中的位置

本系统的估计器不作为一个独立的 Kalman Filter 或粒子滤波器存在，而是嵌入在**感知编排器**（`MagneticCablePerception`，下文简称 orchestrator）中的一组在线修正机制。它的核心任务是：

> 在名义航线先验（nominal route prior）存在平移、旋转、缩放偏差的情况下，利用声呐/局部路径观测在线修正先验，使控制器消费的 route cache 逐步逼近真实电缆位置。

估计器的输入与输出：

```
输入：
  - nominal_route_prior_base_xy        （名义航线先验，已施加初始平移/旋转/缩放噪声）
  - perception.estimated_cable_point_xy_m  （声呐/局部路径观测的电缆点）
  - perception.sonar_heading_deg / local_path_heading_deg  （观测航向）
  - perception.sonar_confidence        （观测置信度）
  - prior_rotation_drift_deg           （独立的先验旋转游走通道，与 DR/INS heading drift 解耦）

输出：
  - nominal_route_prior_correction_translation_xy_m  （累积平移修正量）
  - nominal_route_prior_correction_rotation_deg      （累积旋转修正量）
  - nominal_route_xy                    （修正后的 route cache，供控制器消费）
  - nominal_route_lookup                （投影缓存）
  - last_nominal_route_progress_m       （progress guard 的进度锚点）
```

估计器在每帧 [orchestrator.update()](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/perception/orchestrator.py#L541-L562) 顶部被调用：先按当前车体位置推进 progress guard 锚点，再用本帧声呐读数更新平移/旋转修正。

**控制器侧的接入**：控制器通过 [`ZigZagController.attach_perception()`](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/controller.py#L62-L64) 在仿真初始化时绑定 orchestrator（见 [main_viz.py#L574](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/main_viz.py#L574)），随后 `_nominal_route_reference()` 优先调用 [`orchestrator.nominal_route_projection()`](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/perception/orchestrator.py#L237-L263)，让控制器与感知层共用同一份修正后 + 受 progress guard 约束的 route cache，避免双方各自维护一份导致投影不一致。

> **历史变迁**：早期版本中估计器代码直接挂在 `ZigZagController` 上。引入 EKF prior correction + DR/INS 双坐标系后，由于估计器需要消费声呐 reading 与 navigation pose（两者天然属于感知层），把估计器搬到 orchestrator 既消除了控制器对 sensor_model 的间接依赖，也让 progress guard 的锚点能在感知第一帧就被推进，不必等到控制器进入 update。

### 1.2 估计器组件总览

当前估计器由以下组件构成：

| 组件 | 作用 | 激活条件 |
| --- | --- | --- |
| **Route Prior Correction (gain 模式)** | 用指数平滑修正先验平移+旋转偏差 | `nominal_route_prior_observation_correction_enabled=True` 且 `nominal_route_prior_correction_ekf_enabled=False` |
| **Route Prior Correction (EKF 模式)** | 用 3-D 卡尔曼滤波融合声呐观测与过程噪声 | `nominal_route_prior_correction_ekf_enabled=True` |
| **Residual Gate** | 观测残差过大时拒绝修正 | `max_residual_m > 0`（仅 gain 模式） |
| **Step Gate** | 单步修正量有上限，防止大步跳变 | `max_step_m > 0`（**仅 gain 模式**；EKF 路径不做 step clamp，详见 §1.3.5 lesson） |
| **Heading Error Gate** | heading 误差过大时不更新旋转修正 | `max_heading_error_deg > 0`（仅 gain 模式） |
| **Progress Guard** | 限制 route projection 在进度窗口内搜索 | `nominal_route_progress_guard_enabled=True` |
| **Prior Rotation Drift Channel** | 独立的先验旋转游走通道，与 DR/INS heading drift 解耦 | `navigation.prior_rotation_random_walk_std_deg_per_sqrt_s > 0` |
| **DR/INS Navigation** | 导航位姿与真值分离，模拟慢漂 | `navigation.enabled=True` |
| **Coordinate Consistency** | 声呐相对观测用 navigation pose 转 NED | 自动（启用 DR/INS 时） |

### 1.3 Route Prior Correction：先验路径在线修正

#### 1.3.1 数据流

先验修正的核心数据流如下：

```
                    ┌──────────────────────────────┐
                    │  nominal_route_prior_base_xy  │  ← 初始先验（含噪声）
                    └──────────────┬───────────────┘
                                   │
                    ┌──────────────▼───────────────┐
                    │  apply_route_prior_pose_error │  ← 叠加累积修正量
                    │  (translation + rotation)     │
                    └──────────────┬───────────────┘
                                   │
                    ┌──────────────▼───────────────┐
                    │     nominal_route_xy          │  ← 控制器消费的 route cache
                    └──────────────────────────────┘
                                   ▲
                                   │ 每帧更新
                    ┌──────────────┴───────────────┐
                    │  _update_nominal_route_prior_ │
                    │  observation_correction()     │
                    │                               │
                    │  输入: perception 观测         │
                    │  输出: correction_translation  │
                    │        correction_rotation     │
                    └──────────────────────────────┘
```

#### 1.3.2 入口条件

修正仅在以下条件全部满足时执行（见 [orchestrator.py#L271-L298](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/perception/orchestrator.py#L271-L298)）：

1. `use_nominal_route_prior=True` — 启用了名义先验
2. `nominal_route_prior_observation_correction_enabled=True` — 显式开启了观测修正
3. `sonar_reading is not None and sonar_reading.valid` — 本帧有有效声呐读数
4. `sonar_reading.estimated_position_ned_m is not None` — 已转换到 NED（DR/INS 启用时用 navigation pose，见 §1.6）
5. `sonar_reading.confidence >= nominal_route_prior_correction_min_confidence` — 置信度足够

#### 1.3.3 平移修正（带 Residual Gate + Step Gate）

```python
# 1. 计算观测残差
observed_point_xy = perception.estimated_cable_point_xy_m
prior_point_xy, prior_tangent_xy, residual_distance_m = _nearest_nominal_route_projection(observed_point_xy)

# 2. Residual Gate：残差过大则拒绝
if max_residual_m > 0 and residual_distance_m > max_residual_m:
    return  # 拒绝本次修正

# 3. 计算修正步长（gain 控制修正速率）
residual_xy = observed_point_xy - prior_point_xy
correction_step_xy = gain * residual_xy

# 4. Step Gate：单步修正量上限
if correction_step_norm_m > max_step_m > 0:
    correction_step_xy *= max_step_m / correction_step_norm_m  # 截断

# 5. 累积修正量
nominal_route_prior_correction_translation_xy_m += correction_step_xy

# 6. 累积修正量上限
if correction_norm_m > max_translation_m > 0:
    nominal_route_prior_correction_translation_xy_m *= max_translation_m / correction_norm_m
```

**关键参数**（见 [config/__init__.py#L499-L505](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/config/__init__.py#L499-L505)）：

| 参数 | 默认值 | 含义 |
| --- | ---: | --- |
| `nominal_route_prior_correction_gain` | `0.08` | 修正速率（指数平滑系数），越小越慢越稳 |
| `nominal_route_prior_correction_max_residual_m` | `18.0` | 残差门限，超过此距离的观测视为离群点 |
| `nominal_route_prior_correction_max_step_m` | `0.75` | 单步修正上限，防止单帧大跳 |
| `nominal_route_prior_correction_max_translation_m` | `80.0` | 累积平移修正的硬上限 |
| `nominal_route_prior_correction_min_confidence` | `0.35` | 观测置信度下限 |

#### 1.3.4 旋转修正（带 Heading Error Gate）

```python
# 1. 提取观测航向（声呐优先，其次局部路径）
observed_heading_deg = perception.sonar_heading_deg or perception.local_path_heading_deg

# 2. 计算 heading 误差
prior_heading_deg = heading_from_direction_xy(prior_tangent_xy)
heading_error_deg = smallest_angle_error_deg(observed_heading_deg, prior_heading_deg)

# 3. Heading Error Gate：误差过大则不更新旋转
if max_heading_error_deg > 0 and abs(heading_error_deg) > max_heading_error_deg:
    pass  # 跳过旋转修正
else:
    nominal_route_prior_correction_rotation_deg += gain * heading_error_deg

# 4. 累积旋转修正量上限
nominal_route_prior_correction_rotation_deg = clip(
    nominal_route_prior_correction_rotation_deg,
    -max_rotation_deg, max_rotation_deg
)
```

**关键参数**（见 [config/__init__.py#L503-L505](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/config/__init__.py#L503-L505)）：

| 参数 | 默认值 | 含义 |
| --- | ---: | --- |
| `nominal_route_prior_correction_max_heading_error_deg` | `18.0` | heading 误差门限 |
| `nominal_route_prior_correction_max_rotation_deg` | `8.0` | 累积旋转修正的硬上限 |

#### 1.3.5 EKF Prior Correction（推荐用于 sparse / dropout + DR/INS）

gain 模式（指数平滑 + 三重门限）在静态先验偏差下表现良好，但当 DR/INS 慢漂叠加迷宫回折时存在两个结构性问题：

1. gain 是固定的，无法表达"刚启动时偏差大、收敛后偏差小"的随时间变化的置信度。
2. 门限是硬阈值，无法在"残差略超门限但与累积修正方向一致"的场景中部分接受观测。

EKF 模式将平移与旋转修正抽象成一个 3-D 状态 $\mathbf{x} = (t_x, t_y, r_\theta)$ 与对角协方差 $\mathbf{P}$，每帧做标准的 predict + update（见 [orchestrator.py#L281-L334](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/perception/orchestrator.py#L281-L334)）：

```python
# Predict: 协方差按过程噪声方差时序累加
P[0,0] += (translation_process_std_m_per_sqrt_s ** 2) * dt_s
P[1,1] += (translation_process_std_m_per_sqrt_s ** 2) * dt_s
P[2,2] += (rotation_process_std_deg_per_sqrt_s    ** 2) * dt_s

# Update: 每个观测维度独立做标量 Kalman update
translation_meas_var = (meas_std_m / confidence) ** 2
for axis in (0, 1):
    K = P[axis,axis] / (P[axis,axis] + translation_meas_var)
    x[axis]   += K * innovation_xy[axis]
    P[axis,axis] = (1 - K) * P[axis,axis]
# 旋转维度同理
```

**关键参数**：

| 参数 | 默认值 | 含义 |
| --- | ---: | --- |
| `nominal_route_prior_correction_ekf_enabled` | `False` | 是否启用 EKF 模式 |
| `nominal_route_prior_correction_ekf_initial_translation_var_m2` | `25.0` | 初始平移协方差 |
| `nominal_route_prior_correction_ekf_initial_rotation_var_deg2` | `9.0` | 初始旋转协方差 |
| `nominal_route_prior_correction_ekf_translation_meas_std_m` | `1.5` | 平移观测噪声标准差 |
| `nominal_route_prior_correction_ekf_rotation_meas_std_deg` | `2.5` | 旋转观测噪声标准差 |
| `nominal_route_prior_correction_ekf_translation_process_std_m_per_sqrt_s` | `0.04` | 平移过程噪声（驱动慢漂跟踪） |
| `nominal_route_prior_correction_ekf_rotation_process_std_deg_per_sqrt_s` | `0.025` | 旋转过程噪声 |

**Lesson — 不要在 EKF update 末尾加 step clamp**：

实测中曾尝试在 Kalman update 之后再对 `K * innovation` 做 `np.clip(step, -max_step_m, max_step_m)`，期望"借用 gain 模式的 Step Gate 防御单帧大噪声"。结果在 sparse_mid_prob015 锚点上：

| 指标 | clamp 前 | clamp 后 |
| --- | ---: | ---: |
| health | 84 | 35 |
| route 完成度 | 99.5% | 62% |
| final 横偏 | 0.9m | 512m |

原因：EKF 在**首帧**就期望吸收大部分初始偏差（这是它优于 gain 模式的核心机制——`P` 大、`K → 1`），而 step clamp 把首帧吸收量截到 `max_step_m`，导致估计器从此一直处于"残差大 + 步长被截 + 协方差缓慢膨胀"的状态，最终被进度推到错误 lane。

**结论**：EKF 路径**不要**与 Step Gate 共存。Step Gate 是 gain 模式专属的、用于补偿"gain 固定 + 无置信度建模"的工具；EKF 用 `meas_std / confidence` 的方差缩放已经天然处理了噪声。

### 1.4 Progress Guard：进度连续性保护

#### 1.4.1 问题背景

在迷宫（serpentine）电缆场景中，电缆存在多条平行的 lane 和 U 型折返。当 AUV 在 lane N 上跟踪时，如果使用**全局最近点投影**，route projection 可能跳到相邻 lane N+1 或 N-1 上，导致：

- `route_progress_max_jump_m` 出现数百米级跳变
- `maze_geometry_passed = no`（几何失败）
- 控制器消费了错误的切向和横偏，把车带偏

#### 1.4.2 实现

Progress Guard 限制 route projection 只在上一帧进度附近的窗口内搜索（见 [orchestrator.py#L237-L263](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/perception/orchestrator.py#L237-L263)）：

```python
def nominal_route_projection(self, position_xy_m):
    if not progress_guard_enabled or last_nominal_route_progress_m is None:
        return nearest_point_on_polyline(position_xy_m, nominal_route_lookup)  # 全局搜索

    # 局部窗口搜索
    progress_min_m = last_nominal_route_progress_m - lookback_m
    progress_max_m = last_nominal_route_progress_m + lookahead_m
    return nearest_point_on_polyline_within_progress(
        position_xy_m, nominal_route_lookup, progress_min_m, progress_max_m,
    )
```

局部窗口内的搜索逻辑由 [`nearest_point_on_polyline_within_progress`](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/math_utils.py) 实现：

1. 筛选累积弧长落在 `[progress_min, progress_max]` 区间内的 route segment。
2. 对每个候选 segment 计算最近点，但将投影参数裁剪到窗口边界内。
3. 取距离最小的候选点作为结果。

**锚点推进**：guard 窗口的锚点 `last_nominal_route_progress_m` 由 [`advance_nominal_route_progress()`](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/perception/orchestrator.py#L265-L269) 在每帧 [`update()`](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/perception/orchestrator.py#L558-L561) 顶部按当前车体位置推进，确保第一帧声呐观测就能落入有意义的窗口；EKF / gain 修正成功的末尾也会再刷新一次锚点 + 重建 `nominal_route_lookup`（见 [orchestrator.py#L351-L353](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/perception/orchestrator.py#L351-L353)）。

**关键参数**：

| 参数 | 默认值 | 含义 |
| --- | ---: | --- |
| `nominal_route_progress_guard_enabled` | `False` | 是否启用进度保护 |
| `nominal_route_progress_guard_lookback_m` | `8.0` | 向后搜索窗口 |
| `nominal_route_progress_guard_lookahead_m` | `30.0` | 向前搜索窗口 |

#### 1.4.3 使用策略

Progress Guard **不是全局默认开启的**。当前策略：

| 场景类型 | progress guard | 窗口 | 原因 |
| --- | --- | --- | --- |
| continuous sonar | **关闭** | — | 连续声呐观测足够强，全局投影不会跨 lane；额外约束反而限制正常纠偏（实测开 guard 反而拉低 health） |
| sparse sonar + DR/INS | **开启** | `8 / 30 m` | 稀疏锚点 + 慢漂 + 迷宫回折 = 跨 lane 跳变高风险 |
| dropout-prior + DR/INS | **开启** | `8 / 30 m` | 声呐离线后只有先验，跨 lane 风险高 |

> 历史上 sparse 系列曾使用 `5 / 20 m` 较窄窗口，但在 mid DR/INS + 迷宫 U 型回折时，窗口内候选 segment 不足以覆盖一次 U 弯的弧长，导致投影被锁死在 U 弯起点；扩到 `8 / 30 m` 后既能容纳一次完整 U 弯，又仍小于 lane 间距，避免误投影到相邻 lane。

### 1.5 DR/INS 导航融合

#### 1.5.1 导航位姿与真值分离

系统维护两套位姿：

- **true pose**：仿真真值，用于推进车辆运动学、计算真实磁场。
- **navigation pose**：DR/INS 输出的导航位姿，含慢漂噪声，用于感知和控制。

分离的关键意义：控制器和感知层只能看到 navigation pose，不能访问 true pose。这模拟了真实部署中 GPS/INS 存在漂移的情况。

#### 1.5.2 NavigationSimulator：慢漂发生器

`NavigationSimulator`（见 [sensor_model.py#L418-L496](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/sensor_model.py#L418-L496)）模拟 DR/INS 的典型误差特性：

```
navigation_position = true_position + drift + filtered_noise
navigation_heading  = true_heading  + drift + filtered_noise
```

噪声模型包含三个层次：

1. **白噪声**（`position_white_noise_std_m`、`heading_white_noise_std_deg`）：逐帧独立的高斯噪声。
2. **低通滤波**（`lowpass_alpha`）：对白噪声做一阶低通，模拟惯导的慢变误差特性。`alpha` 越小，滤波越强，噪声越"粘稠"。
3. **随机游走漂移**（`position_random_walk_std_m_per_sqrt_s`、`heading_random_walk_std_deg_per_sqrt_s`）：随时间累积的零均值随机游走，模拟惯导积分误差。

硬上限保护：

- `max_position_drift_m`：位置漂移的硬上限（默认 `80m`）。
- `max_heading_drift_deg`：航向漂移的硬上限（默认 `8°`）。

#### 1.5.3 预置漂移档位

为方便调参，预置了三档 DR/INS 漂移强度（见 [config/__init__.py#L1355-L1369](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/config/__init__.py#L1355-L1369)）：

| profile | pos white noise | pos random walk | hdg white noise | hdg random walk | 适用场景 |
| --- | ---: | ---: | ---: | ---: | --- |
| `mild` | `0.15m` | `0.003 m/√s` | `0.15°` | `0.001 °/√s` | 高质量 INS / DVL 辅助 |
| `mid` | `0.20m` | `0.005 m/√s` | `0.20°` | `0.001 °/√s` | 中等质量 INS |
| `strong` | `0.35m` | `0.018 m/√s` | `0.35°` | `0.003 °/√s` | 低质量 INS / 纯 DR |

#### 1.5.4 Prior Rotation Drift：与 DR/INS 解耦的先验旋转游走通道

DR/INS 的 heading drift 与"操作员提供的名义路径先验的旋转误差"是两个独立的物理量：

- DR/INS heading drift 影响**车体航向的测量值**，所有姿态/控制都基于这个有偏 heading。
- Prior rotation drift 影响**先验路径相对于真实电缆的旋转**——例如海图录入误差或电缆敷设方向标定漂移导致的先验角度偏差。

把这两个通道混在一起会导致估计器无法区分"航向被测错"和"先验被画歪"——两者对声呐残差的贡献方向相同但需要不同的修正策略（前者由 INS 闭环修正，后者由 prior correction 修正）。

实现：[`NavigationSimulator`](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/sensor_model.py#L419-L510) 维护独立的 `prior_rotation_drift_deg` 字段，按 `prior_rotation_random_walk_std_deg_per_sqrt_s` 做随机游走，封顶 `max_prior_rotation_drift_deg`（默认 `4°`），通过 [`NavigationMeasurement.prior_rotation_drift_deg`](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/sensor_model.py#L55) 注入到 orchestrator。Orchestrator 在 [`update()`](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/perception/orchestrator.py#L555-L557) 中检测到该字段变化时刷新 route lookup，并在 [`_refresh_nominal_route_lookup()`](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/perception/orchestrator.py#L223-L235) 中把它叠加到 `nominal_route_prior_correction_rotation_deg` 之外的"effective rotation"上——即 prior rotation drift 是估计器**需要持续追赶**的真实未知量，而 `correction_rotation_deg` 是估计器**已经追赶到的累积修正**。

**关键参数**：

| 参数 | 默认值 | 含义 |
| --- | ---: | --- |
| `navigation.prior_rotation_random_walk_std_deg_per_sqrt_s` | `0.0` | 旋转游走过程噪声 |
| `navigation.max_prior_rotation_drift_deg` | `4.0` | 旋转游走硬上限 |

### 1.8 先验形变（Scale Distortion）

除了平移和旋转，系统还支持对名义先验施加缩放形变（见 [math_utils.py#L441-L463](file:///Users/bytedance/coding/AUV-Master-Mag/src/auv_mag_tracking/math_utils.py#L441-L463)）：

```python
def apply_route_prior_pose_error(route_xy, translation, rotation, scale_xy):
    # 变换顺序：缩放 → 旋转 → 平移
    transformed = (route - pivot) * scale + pivot       # 缩放
    transformed = (transformed - pivot) @ rotation.T + pivot  # 旋转
    return transformed + translation                    # 平移
```

当前测试使用 `scale_xy = (0.99, 1.0)`，模拟先验在沿电缆方向有 1% 的压缩。这是一个保守的形变量，代表"先验本身存在轻微扭曲（虽然很小）"的工程假设。

### 2.3 门限参数调优指南

#### 2.3.1 correction_gain（修正速率）

| 值 | 行为 | 适用场景 |
| ---: | --- | --- |
| `0.01` | 极慢修正，约 100 帧才收敛到残差的 63% | DR/INS 慢漂场景（推荐） |
| `0.08` | 中等修正 | 静态先验偏差场景 |
| `0.20` | 较快修正 | 仅用于大偏差快速收敛，风险较高 |

**调参原则**：先验修正必须"慢"。快的修正会放大单帧观测噪声，在稀疏锚点场景下尤其危险。DR/INS 慢漂本身是缓慢变化的，慢修正足以跟踪。

#### 2.3.2 max_residual_m（残差门限）

残差门限的作用是**拒绝离群观测**。当声呐观测点与当前先验投影点的距离超过此门限时，该帧观测被丢弃。

| 值 | 行为 |
| ---: | --- |
| `0` | 关闭残差门限（不推荐） |
| `18.0` | 默认值，适合大多数场景 |
| `10.0` | 更保守，适合高噪声场景 |

**调参原则**：门限应大于"正常先验偏差 + 观测噪声"之和，但小于"跨 lane 距离"。在迷宫场景中，lane 间距为 `60m`（或 `100m`），`18m` 远小于 lane 间距，能有效拒绝跨 lane 的误匹配。

#### 2.3.3 max_step_m（单步修正上限）

| 值 | 行为 | 适用场景 |
| ---: | --- | --- |
| `0.75` | 默认值 | continuous sonar、dropout-prior |
| `0.35` | 收紧 | sparse sonar + DR/INS |
| `0.15` | 极紧 | 极高噪声或极稀疏锚点 |

**调参原则**：单步修正上限应与"相邻帧之间先验漂移量"匹配。DR/INS 慢漂在单帧（`dt=0.05s`）内的漂移量远小于 `0.35m`，因此 `0.35m` 的上限不会阻碍正常修正，但能有效阻止单帧异常观测造成的大跳。

#### 2.3.4 max_heading_error_deg（heading 误差门限）

| 值 | 行为 | 适用场景 |
| ---: | --- | --- |
| `18.0` | 默认值 | continuous sonar、dropout-prior |
| `12.0` | 收紧 | sparse sonar + DR/INS |

**调参原则**：heading 误差门限应大于"先验旋转偏差 + 观测噪声"之和。在 `3°` 旋转先验偏差 + `5°` 声呐 heading 噪声的场景中，`12°` 的门限有足够余量。

| 硬件 | 推荐 profile | 说明 |
| --- | --- | --- |

## 3. 设计理念（面向论文写作）

### 3.1 核心设计哲学：慢速、受门限保护、局部窗口

本估计器的设计遵循一个核心原则：

> **估计器修正必须"慢速、受门限保护、局部窗口"。全局最近点匹配在迷宫场景下不可接受。**

这一原则源于以下观察：

1. **声呐观测是稀疏且带噪声的**。在 sparse sonar 场景中，`prob_detection=0.20` 意味着 80% 的帧没有观测。如果修正过快，少数几帧噪声观测就能把先验拉偏。
2. **DR/INS 慢漂是缓慢变化的**。在 `dt=0.05s` 的时间尺度上，单帧漂移量远小于观测噪声。慢修正足以跟踪漂移，快修正只会放大噪声。
3. **迷宫几何存在多 lane 歧义**。全局最近点匹配在回折处会跳到相邻 lane，导致控制器消费错误的切向和横偏。

### 3.2 为什么需要 Progress Guard（而非全局匹配）

#### 3.2.1 问题形式化

设电缆路线为一条有自交（回折）的折线 $\mathcal{R}$，AUV 当前位置为 $\mathbf{p}$。全局最近点投影：

$$\mathbf{p}^* = \arg\min_{\mathbf{r} \in \mathcal{R}} \|\mathbf{p} - \mathbf{r}\|$$

在迷宫场景中，$\mathcal{R}$ 包含多条平行的 lane。当 AUV 在 lane $k$ 上时，$\mathbf{p}^*$ 可能落在 lane $k+1$ 或 $k-1$ 上，因为平行 lane 之间的几何距离可能小于 AUV 到当前 lane 上"正确"投影点的沿程距离。

#### 3.2.2 解决方案：进度窗口约束

Progress Guard 将搜索空间从全局 $\mathcal{R}$ 限制为进度窗口内的局部子集：

$$\mathcal{R}_{\text{window}} = \{\mathbf{r}(s) \in \mathcal{R} \mid s \in [s_{\text{last}} - \Delta_{\text{back}}, s_{\text{last}} + \Delta_{\text{forward}}]\}$$

其中 $s$ 是沿电缆的弧长参数，$s_{\text{last}}$ 是上一帧的投影进度。

窗口大小由两个参数控制：

- $\Delta_{\text{back}}$（`lookback_m`）：向后搜索窗口，默认 `8m`。
- $\Delta_{\text{forward}}$（`lookahead_m`）：向前搜索窗口，默认 `30m`。

窗口的非对称设计（前向窗口远大于后向）反映了 AUV 主要向前运动的事实，同时保留少量后向窗口以容忍局部振荡。

#### 3.2.3 为什么不是全局默认开启

Progress Guard 有一个隐含假设：**上一帧的投影进度是可靠的**。如果初始投影就已经错了（例如初始先验偏差过大导致投影到错误 lane），窗口约束会把错误锁死。

因此，Progress Guard 只在"连续跟踪中可能跨 lane"的场景（sparse sonar、dropout-prior）中开启，而在"观测足够强、初始投影可靠"的场景（continuous sonar）中关闭。

### 3.3 为什么需要多层门限

估计器的三层门限（Residual Gate、Step Gate、Heading Error Gate）各自防护不同的失效模式：

| 门限 | 防护的失效模式 | 理论依据 |
| --- | --- | --- |
| **Residual Gate** | 声呐误检/离群点 | 单帧观测残差超过物理合理范围（如跨 lane），应视为离群点 |
| **Step Gate** | 单帧大跳 | 物理系统在 `dt=0.05s` 内不可能发生米级位移，大修正步长必然是噪声 |
| **Heading Error Gate** | 声呐航向误判 | 声呐在低 SNR 下可能输出 180° 翻转的航向，heading 误差过大的修正会污染旋转估计 |

这三层门限构成了一个**纵深防御体系**：

- Residual Gate 是第一道防线：拒绝明显离群的观测。
- Step Gate 是第二道防线：即使观测通过了残差检查，单帧修正量也被限制。
- Heading Error Gate 是第三道防线：保护旋转估计不被航向噪声污染。

### 3.4 DR/INS 坐标系一致性的设计哲学

启用 DR/INS 后，系统面临一个根本性的坐标系选择问题：

> 声呐观测（相对机体坐标系的电缆位置）转换为 NED 绝对坐标时，应该用 true pose 还是 navigation pose？

**答案：必须用 navigation pose。**

原因：控制器和感知层只能访问 navigation pose。如果用 true pose 转换声呐观测，得到的 NED 电缆点位于"真值坐标系"，而控制器消费的 route prior 和 AUV 位置位于"导航坐标系"。两个坐标系之间的偏差就是 DR/INS 漂移量，这会导致：

1. 估计器残差异常偏大（残差 = 观测噪声 + DR/INS 漂移）。
2. Residual Gate 可能错误地拒绝有效观测。
3. 控制器横偏计算在错误的坐标系中进行。

统一使用 navigation pose 后，所有量都在同一（有偏的）坐标系中，估计器修正的是"先验相对于导航坐标系的偏差"，而非"先验相对于真值的偏差"。这是一个关键的**参照系选择**决策。

### 3.5 先验形变（Scale Distortion）的角色

先验缩放形变 `scale_xy = (0.99, 1.0)` 模拟了一个现实但微妙的误差源：名义航线在沿电缆方向存在 1% 的压缩。这对应以下工程场景：

- 海图投影误差导致的系统性尺度偏差。
- 电缆敷设时的实际长度与设计长度的差异。
- 不同坐标系（经纬度 → UTM）转换时的尺度因子误差。

在当前系统中，缩放形变**不被估计器在线修正**（`_refresh_nominal_route_lookup()` 中 scale 固定为 `(1.0, 1.0)`）。这意味着缩放形变是一个**不可观测的残差**，系统通过以下方式容忍它：

1. 平移修正可以部分补偿局部尺度误差（在短窗口内，尺度误差近似为平移偏差）。
2. Progress Guard 的局部窗口搜索对尺度误差不敏感（窗口内的局部几何几乎不受全局尺度影响）。
3. 声呐观测的优先使用（`_fresh_observation_*`）绕过了先验的尺度问题。

### 3.7 当前边界与未解决问题

#### 已确认的可用边界

下表来自 [results/20260629_prior_tiers/critical_sweep.csv](file:///Users/bytedance/coding/AUV-Master-Mag/results/20260629_prior_tiers/critical_sweep.csv) 6 锚点 sweep 实测：

| 锚点 | 场景 | health | route | endpoint | geometry | max_jump_m | jumps≥10m | 判定 |
| --- | --- | ---: | ---: | --- | --- | ---: | ---: | --- |
| 0 | sparse_mid_prob015 | 84.6 | 99.5% | yes | yes | 0.9 | 0 | ✅ pass |
| 2 | sparse_heavy | 90.2 | 99.5% | yes | yes | 0.4 | 0 | ✅ pass |
| 3 | dropout_mid | 81.8 | 99.5% | yes | yes | 0.2 | 0 | ✅ pass |

这三个锚点构成"当前算法稳定承受"的可重复边界：

- **sparse 系列**：在 sparse + mid/heavy DR/INS + 静态偏差 (7.5–10m, 3–5°) + 旋转游走 + scale 0.99/0.98 下，EKF + Progress Guard 8/30m 配合 `max_step_m=0.20` 能把 max_jump 压到 < 1m，无跨 lane。
- **dropout**：声呐进入 TRACK 后强制离线，残差极少，progress guard 几乎独自支撑投影连续性，1 次大跳变都没有。
- **sparse_mid_prob015**：sonar `prob_detection=0.15` 时仍 pass，说明 EKF + `max_step_m=0.18` + guard 的组合对低可观测性场景具备一定鲁棒性。

#### 已接受为算法边界的失效

| 锚点 | 场景 | max_jump | 大跳变次数 | 物理解释 |
| --- | --- | ---: | ---: | --- |
| 1 | sparse_mid | 67.4m | 1 | 单次 U 弯起点处投影落到相邻 lane；EKF 收敛后仍能压回正轨，但 1 次跨 lane 已破坏 geometry 验收 |
| 4 | continuous sonar mid | ~80m | 2–3 | continuous + 无 guard 下，迁移锚点附近一次跨 lane 投影；与静态偏差大小关系不单调（缩到 1m 反而比 3m 更差） |
| 5 | continuous sonar heavy | ~80m | 2 | 同上，且 `strong` DR/INS 下偶发 final 横偏发散到数百米 |

这三个失效的共同特征是**单次跨 lane 跳变**，而非渐进发散，说明它们是几何/拓扑层面的歧义问题而非门限调参问题：

- sparse_mid 的根因是稀疏锚点 + DR/INS 中漂下 U 弯起点的 progress guard 窗口边界附近存在另一段 segment（lane 间距 ≈ 60m，guard `lookahead=30m`，刚好等于半 lane 距离的临界值附近）。再扩大 lookahead 会反过来把 sparse_heavy 拉退化。
- continuous sonar mid/heavy 的根因是 progress guard 被关闭后全局投影对 prior rotation drift × DR position drift 的耦合误差不鲁棒。把静态偏差从 7.5m 缩到 3m 再缩到 1m 期间，jump 一度从 335m → 77m，但继续缩反而恶化（362m / 501m final），说明 continuous + 无 guard 下大 jump 与"静态偏差大小"非单调相关，单纯压偏差不是解。

#### 真正需要新增的机制

| 失效类型 | 建议机制 |
| --- | --- |
| sparse 单次 U 弯跨 lane | lane-aware projection guard：基于电缆几何拓扑而非弧长距离的窗口；或 sparse anchor confidence memory，把 TRACK 阶段未漂的锚点保留为投影偏好 |
| continuous + 无 guard 大 jump | 在 continuous 场景下重新设计"软 guard"：不强制窗口约束，但对跨 lane 候选的投影距离加惩罚项 |
| sparse_mid_prob015 安全边际 | 与 magnetic path 局部拓扑约束融合，把磁路径与 prior 投影做一致性校验后再吸收 |

这些都属于**架构性变更**而非门限调参，预计需要单独立项。
