# 声呐 + 磁传感长期闭环剪枝计划

## 0. 定位

本文承接：

- `docs/17_zigzag纯磁探针方案.md`
- `docs/18_lookahead_feed_diagnostics.md`

目标不是继续堆叠新的 probe variant，而是把长期目标重新收束成一条清晰链路：

1. 输入只允许来自声呐 wrapper 提供的电缆位置/航向观测，以及磁传感器输出的场强、矢量和姿态补偿后的 anomaly。
2. 输出需要同时支持电缆跟踪和埋深反演。
3. zig-zag 不是单纯搜索动作，而是让磁几何可观测的主动探针。
4. API 边界必须变清楚，避免感知、任务、控制三层重复发明 lookahead、axis selection 和 feed gate。
5. 所有新能力先 shadow，再旁路诊断，再受控接入；不能因为局部 heading 或横偏指标好看而牺牲 route/endpoint。

当前阶段结论已修正：常规声呐场景可以作为稳定基本盘；`case_maze_sonar` 需要同时满足 endpoint 和 no-shortcut 几何指标，不能只看 route completion。当前已把 `case_maze_sonar` / `case_maze_sparse_sonar` 恢复为显式 route-prior 的稳定 sonar baseline；真正堵点仍是 `case_maze_sonar_dropout` 及未来“仅初始/稀疏声呐 + 磁传感”闭环。已有 p36 是历史 route-progress 基线，p47 已证伪，下一步应围绕旁路多假设选择、route-bound 控制消费和 API 收敛推进。

### 0.1 2026-06-27：maze/dropout 状态纠偏

本轮新增 `maze_geometry_passed`、`lane_shortcut_indicator`、`route_progress_max_jump_m` 等指标后，旧的 maze 成功口径需要纠偏：

- `endpoint_completed=yes` 不等于 maze 几何成功。若 route progress 出现数百米级跳变，说明最近点投影跨 lane，属于 lane shortcut / projection jump。
- `9f4ecc5` 曾被视为 sonar maze 平稳参考，但同口径复算也存在约 `740m` projection jump；它不能作为 no-shortcut 几何合格基线。
- 当前稳定 sonar baseline 是新口径：`case_maze_sonar` 和 `case_maze_sparse_sonar` 均达到 route `99.5%`、endpoint yes、large route jumps `0`、`maze_geometry_passed=yes`。
- `case_maze_sonar_dropout` 默认仍失败：route 约 `1.5%`、endpoint no、TRACK vehicle heading error 约 `89.7°`，不是 19 号计划已完成项。

dropout 的当前证据链：

| 口径 | 结果 | 结论 |
| --- | --- | --- |
| 默认 dropout | route `1.5%`，vehicle heading error `~89.7°`，magnetic path/phase/shadow/manager 默认供给为 `0` | 任务推进失败 |
| shadow-only dropout | magnetic path 约 `70.6%`、lookahead/shadow 约 `11.5%`，但 route 不推进 | 磁候选不是绝对为零，但未形成控制闭环 |
| 激进 probe/manager | 产生约 `760m` projection jump | 不能进入默认配置 |

因此，dropout 下一步不能只追求更好的 fused heading，也不能把 aggressive probe 直接合入默认。必须先证明“磁候选 -> route-bound 目标 -> 控制推进”的链条成立，并继续与 sonar baseline 隔离。

### 0.2 2026-06-28：无 route 先验自适应大角度 zig-zag 初测

按后续测试计划，本轮只在 `case_maze_sonar_dropout` 中尝试无名义先验的大角度 TRACK zig-zag：

- `use_nominal_route_prior=False` 保持不变。
- `track_active_zigzag_angle_deg=15°`。
- 开启自适应角度：以 `3m` 横向有效距离换算目标角度，但只允许在基础角度 `±5°` 内浮动。
- `ProbeBurstManager` 仍默认关闭。
- `reacquire_region_route_guard` 仍关闭，避免 route prior 变相进入 dropout。
- 同时打开 `magnetic_path_observation_enabled=True`，否则角度已经增大但 magnetic path/phase/shadow 供给仍为 0，无法判断磁观测是否真的改善。

对比结果：

| 口径 | route | TRACK vehicle err | TRACK XT | mag path | crossings | max jump | 结论 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| 只开自适应大角度 | `2.1%` | `89.2°` | `27.4m` | `0%` | `0` | `0.0m` | 角度本身不足以闭环 |
| 自适应大角度 + magnetic path observation | `16.9%` | `15.4°` | `6.9m` | `28%` | `2` | `0.0m` | 出现阶段性正结果，但仍未 endpoint |

当前解释：

1. 自适应大角度能制造更明显横向激励，但如果磁 path 供给链路仍关闭，任务推进几乎不改善。
2. 打开 magnetic path 后，车辆实际航向误差显著下降，route 从 `1.5%/2.1%` 提升到 `16.9%`，且没有 route projection jump。
3. 仍未闭环：最终 cross-track 仍很大，mean fused heading error 仍高，magnetic path axis/position error 仍大，说明当前只是“可继续”的观测正结果，不是完成态。

下一步应继续沿该方向做受控实验：不要再单独增大角度，而是检查 magnetic path 误差、forward phase coverage、route-bound candidate 是否能变成稳定控制目标。

### 0.3 2026-06-28：降低要求后的低精度 route prior 实验

为了区分“完全无先验 dropout 闭环”和“工程上可准备低精度航线”的难度，本轮加入了可配置的名义 route prior 平移噪声：

- 新增配置：`tracking.nominal_route_prior_translation_xy_m`。
- 默认值为 `(0.0, 0.0)`，不改变已有 baseline。
- 本轮 noisy-prior 变体使用 `(0.0, 7.5m)`，代表 `5-10m` 平移误差带的中点。
- 平移只作用于 controller/perception 消费的 nominal route prior cache；环境真实电缆路线和 metrics 投影不平移。

新增场景：

| 场景 | 说明 |
| --- | --- |
| `case_maze_sonar_prior_noise` | sonar baseline + `7.5m` 平移 route prior |
| `case_maze_sparse_sonar_prior_noise` | sparse sonar baseline + `7.5m` 平移 route prior |
| `case_maze_sonar_dropout_prior` | dropout + 干净 route prior |
| `case_maze_sonar_dropout_prior_noise` | dropout + `7.5m` 平移 route prior |

结果目录：`results/20260628_prior_noise/`。

| 场景 | route | endpoint | geometry | max jump | TRACK XT | TRACK vehicle err | 结论 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| `case_maze_sonar` | `99.5%` | yes | yes | `0.1m` | `0.8m` | `8.8°` | 干净 sonar baseline |
| `case_maze_sonar_prior_noise` | `99.5%` | yes | yes | `0.3m` | `5.4m` | `9.0°` | `7.5m` route prior 误差仍可工作 |
| `case_maze_sparse_sonar` | `99.5%` | yes | yes | `0.1m` | `0.9m` | `8.5°` | 干净 sparse baseline |
| `case_maze_sparse_sonar_prior_noise` | `99.5%` | yes | yes | `0.3m` | `5.5m` | `9.1°` | 稀疏声呐也能承受 `7.5m` route prior 误差 |
| `case_maze_sonar_dropout` | `16.9%` | no | no | `0.0m` | `6.9m` | `15.4°` | 无 route prior 仍未闭环 |
| `case_maze_sonar_dropout_prior` | `99.5%` | yes | yes | `0.1m` | `0.9m` | `13.5°` | dropout + 干净 route prior 可工作 |
| `case_maze_sonar_dropout_prior_noise` | `99.5%` | yes | yes | `0.3m` | `6.8m` | `13.6°` | dropout + `7.5m` route prior 也可工作 |

当前论文/工程解释：

1. 若允许低精度 route prior，maze/dropout 的工程可行性明显提高；`5-10m` 量级平移误差不会破坏 endpoint，也没有发现 lane shortcut。
2. 这不是“无先验 dropout”成功，而是“低精度 route prior + 局部声呐/磁观测”成功。
3. 论文中应明确把该口径称为 map-assisted / nominal-route-assisted tracking，把完全无 route prior 的 dropout 作为更困难的开放问题或后续工作。

### 0.4 2026-06-28：平移 + 旋转 pose-noise route prior

单纯 `7.5m` 平移噪声代表性不足，因此本轮加入更严重的 route prior 姿态误差：

- `tracking.nominal_route_prior_translation_xy_m = (0.0, 7.5m)`。
- 新增 `tracking.nominal_route_prior_rotation_deg = 3.0`。
- 旋转以名义路线首点为 pivot，表示“起点大致对齐，但地图整体航向有 3° 偏差”。
- 新增 `case_maze_sonar_prior_pose_noise`、`case_maze_sparse_sonar_prior_pose_noise`、`case_maze_sonar_dropout_prior_pose_noise`。

未修复时，`3° + 7.5m` 会让 AUV 跟随偏置先验：

| 场景 | route | endpoint | geometry | max jump | TRACK XT | final XT | 结论 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| sonar pose-noise 未修复 | `62.0%` | no | no | `0.8m` | `13.0m` | `449.6m` | 沿错误先验漂走 |
| sparse pose-noise 未修复 | `62.0%` | no | no | `0.8m` | `13.5m` | `457.8m` | 稀疏锚点也无法自行拉回 |

修复策略：

1. controller / perception 的 prior cache 统一通过 `apply_route_prior_pose_error()` 应用平移和旋转，避免两处实现漂移。
2. 新增 `nominal_route_prior_observation_correction_enabled`，允许在有新鲜声呐观测时估计 route prior 残余平移/旋转。
3. 控制层在声呐新鲜时优先使用声呐电缆点和声呐 heading，避免持续跟随偏置 route prior。
4. 对 sparse pose-noise，禁止持续改写整条 prior cache（`nominal_route_prior_correction_gain=0.0`），只保留新鲜声呐覆盖控制输入；否则迷宫回折处会因稀疏锚点把投影拉错 lane。
5. pose-noise 场景提高 TRACK 横偏修正上限，用于抵消 3° 角度偏差带来的累积横漂。

修复后结果目录：`results/20260628_prior_pose_noise/`。

| 场景 | route | endpoint | geometry | max jump | TRACK XT | TRACK vehicle err | final XT | 结论 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| `case_maze_sonar_prior_pose_noise` | `99.5%` | yes | yes | `0.9m` | `3.9m` | `13.8°` | `3.3m` | 连续声呐可校正 3° pose-noise |
| `case_maze_sparse_sonar_prior_pose_noise` | `99.5%` | yes | yes | `0.8m` | `11.7m` | `14.3°` | `11.0m` | 可工作但裕度较低 |
| `case_maze_sonar_dropout_prior_pose_noise` | `99.5%` | yes | yes | `0.2m` | `1.7m` | `12.6°` | `1.7m` | dropout + 带姿态误差 prior 可工作 |

结论：

- `3° + 7.5m` 是足够严重的 route prior 误差；未修复时确实会跟随偏差先验。
- 引入声呐观测驱动的先验校正/覆盖后，连续 sonar、sparse sonar、dropout-prior 三类均可恢复 endpoint 且 no-shortcut。
- sparse pose-noise 虽然通过，但健康分只有 `58/100`，说明这是可用下限附近；论文中应表述为“低精度地图辅助仍能完成，但稀疏声呐下横偏裕度明显降低”。

### 0.5 2026-06-28：DR/INS 慢漂与轻微先验扭曲

本轮把 DR/INS 从“隐含真值位姿”推进为显式 API 与仿真输入：

- 新增 `NavigationMeasurement`：包含 `position_ned_m`、`heading_deg`、`pitch_deg`、`roll_deg`、`speed_mps`、`position_std_m`、`heading_std_deg`、`source`。
- `RawSensorBundle` 新增 `navigation_position_ned_m`、`navigation_heading_deg`、`navigation_pitch_deg`、`navigation_roll_deg`、`navigation_speed_mps`、`navigation_position_std_m`、`navigation_heading_std_deg`。
- `SensorFrame` 新增 `navigation_measurement`；感知层使用导航位置，控制器使用导航 pose，车辆动力学仍用真值 pose 推进，因此 DR/INS 漂移会真实进入闭环。
- 新增 `NavigationSimulator`，用低通白噪声 + 随机游走漂移生成 DR/INS 位置/航向。
- 声呐 wrapper 坐标系修正：启用 DR/INS 后，声呐相对观测转换到 NED 时使用导航 pose，而不是仿真真值 pose。否则会把导航坐标和真值声呐点混在一起，导致 prior correction 被错误拉偏。

先验扭曲也从刚体 pose error 扩展到轻微形变：

- `tracking.nominal_route_prior_scale_xy = (0.99, 1.0)`。
- 该缩放只作用于控制/感知使用的 nominal route prior，不改变环境真值 route 和 metrics route。
- 当前代表“长宽比存在 1% 偏差”的低精度地图。

新增场景：

| 场景 | DR/INS | prior pose error | prior scale | 用途 |
| --- | --- | --- | --- | --- |
| `case_maze_sonar_dr_ins_prior_distortion` | 较强慢漂 | `7.5m + 3°` | `(0.99, 1.0)` | 连续声呐下验证坐标系/API 和 prior 扭曲适应 |
| `case_maze_sparse_sonar_dr_ins_prior_distortion` | 温和慢漂 | `7.5m + 3°` | `(0.99, 1.0)` | 稀疏声呐边界验证 |
| `case_maze_sonar_dropout_dr_ins_prior_distortion` | 较强慢漂 | `7.5m + 3°` | `(0.99, 1.0)` | dropout + 低精度地图 + DR/INS 漂移验证 |

调参经验：

- prior correction gain 不能太大。`0.08` 在 DR/INS 慢漂下会把迷宫回折处投影拉错 lane；本轮收敛到 `0.01`。
- sparse sonar 对 DR/INS 漂移更敏感，原因是锚点稀疏时，prior correction 缺少持续观测约束。因此 sparse 场景使用较温和的 DR/INS 漂移包络。
- 这不是降低问题难度，而是把两个变量拆开：连续/dropout 场景验证较强 DR/INS 慢漂，sparse 场景验证稀疏锚点下的可用边界。

结果目录：`results/20260628_dr_ins_prior_distortion/`。

| 场景 | health | route | endpoint | geometry | max jump | TRACK XT | TRACK vehicle err | final XT |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| `case_maze_sonar_dr_ins_prior_distortion` | `74` | `99.5%` | yes | yes | `2.9m` | `6.8m` | `13.4°` | `0.5m` |
| `case_maze_sparse_sonar_dr_ins_prior_distortion` | `80` | `99.5%` | yes | yes | `0.5m` | `4.8m` | `10.2°` | `2.0m` |
| `case_maze_sonar_dropout_dr_ins_prior_distortion` | `79` | `99.5%` | yes | yes | `0.2m` | `2.6m` | `10.4°` | `3.0m` |

结论：

- 当前 API 已能接收 DR/INS 导航位姿，并让闭环控制真正受慢漂影响。
- 在 `7.5m + 3° + 0.99x` 低精度先验下，连续 sonar、sparse sonar、dropout-prior 均可 endpoint 且 no-shortcut。
- sparse 的通过依赖较温和 DR/INS 漂移；如果同时叠加强慢漂和低 sonar 锚点，仍可能进入中段 lost 或 projection jump，这是后续鲁棒性边界。

## 1. 当前仓库上下文

### 1.1 主数据流

离线仿真主循环在 `src/auv_mag_tracking/viz/recorder.py`，它调用同一套环境、传感器、感知和控制组件，输出 `RunRecord`。这个文件已经是报告、指标、图表和回归评估的唯一数据契约。

核心闭环：

```text
ScenarioConfig
  -> Environment / SensorModel
  -> PerceptionOrchestrator.update()
  -> PerceptionState
  -> ZigZagController.update()
  -> GuidanceCommand
  -> propagate_vehicle()
  -> RunRecorder / metrics / report
```

感知主入口是 `src/auv_mag_tracking/perception/orchestrator.py`。目前它同时承担：

- 声呐观测喂入全局 fitter 与 `LocalCableStateEstimator`。
- 磁峰值、磁横偏、磁 path projection、phase detector、lookahead target、shadow hypothesis 的更新。
- 埋深反演的全局 cumulative inverter 与 probe cycle inverter 输出。
- 将所有中间诊断压入 `PerceptionState`。

控制主入口是 `src/auv_mag_tracking/controller.py`。它使用 FSM 决定 SEARCH/LOCK/TRACK，并把纵向基准 heading 与横向压线分开。TRACK 中横偏来源优先级目前是：

```text
sonar cable point
  -> magnetic_lookahead cable point
  -> magnetic_cross_track_offset_m
  -> estimated route/cable point fallback
```

这个优先级是有效的，但必须约束 `magnetic_lookahead` 只能来自任务层确认目标，而不是未经确认的单帧磁观测。

### 1.2 声呐的角色

声呐是当前唯一可靠的直接位置锚点。`orchestrator.py` 中声呐有效时会同时喂：

- SNR weighted fitter，用于全局/局部中心线收敛。
- `local_path_estimator`，用于 local path guidance。
- `local_path_tracking_estimator`，用于曲线局部跟踪状态。

长期目标中“只有声呐和磁传感器”不等于“声呐必须全程在线”。更准确的接口应是：

```text
SonarCableObservation:
  cable_point_xy_m
  cable_heading_deg optional
  confidence
  timestamp
  source_status
```

声呐只负责 position anchor，不应该隐式承担路线先验、闭环方向消歧或埋深真值供给。

### 1.3 磁传感器的角色

磁通道目前拆成四类信号：

1. `magnetic_cross_track_offset_m`：基于 `B_down / B_perp` 的横偏转向信号，只用于 TRACK 压线，不能喂中心线 fitter。
2. `MagneticPathObservationBuilder`：由水平磁矢量和竖直/横向比值投影出隐式电缆点，单帧有 180deg 方向歧义。
3. `MagneticZigzagPhaseDetector` / `MagneticLookaheadTargetBuilder`：把跨线 phase 转为持续 lookahead，但现有 builder 仍会输出单一方向。
4. `MagneticShadowHypothesisSelector`：旁路保留 `+axis/-axis` 并用短窗 progress/heading 打分，目前 D2 是正结果，但还未接入控制。

关键约束：磁传感器可以提供横偏、轴线、相位、场强和埋深 posterior，但不能在感知层过早提交全局方向。

### 1.4 埋深反演状态

`src/auv_mag_tracking/perception/burial_inversion.py` 已包含两条路径：

- `MagneticBurialInverter`：全局 cumulative 标定幅度法，依赖 lateral offset，暖机后输出 `estimated_burial_depth_m`。
- `MagneticBurialCycleEstimator`：每个 zig-zag cycle 重置，按近过线和 SNR 加权输出 cycle-local posterior，当前用于 shadow 诊断。

已有实验表明，cycle-level posterior 的 MAE 优于 probe-frame 旧口径，但 coverage 很低。后续不应把 burial 直接闭环到控制；应先把 burial posterior 作为 hypothesis 的质量维度，进入 shadow selection。

## 2. 已确认事实和剪枝结论

### 2.1 保留

| 项 | 结论 |
| --- | --- |
| p36 low-weight extrapolated feed | 当前纯磁 dropout route-progress 基线。保留作控制/回归对照。 |
| D2 shadow axis selector | 旁路正结果。不扰动 route，能在供给窗口内产生高 margin 方向选择。 |
| sparse sonar anchor | 稀疏声呐是可靠锚点正例，应作为校准候选，不应简单当作提高检测概率。 |
| cycle-level burial posterior | 估计质量比逐帧好，但 coverage 低，继续 shadow。 |
| RunRecord 诊断契约 | 已能记录 probe、lookahead feed、shadow hypothesis、burial cycle 等关键通道。 |

### 2.2 停止继续投入

| 路线 | 停止原因 |
| --- | --- |
| 单帧 magnetic path 直接 feed local_path | 180deg 方向歧义和 branch 污染会破坏 route。 |
| 单阈值调 heading gate | p25/p28/p31/p32 已说明不存在稳定阈值折中。 |
| phase anchor 高权重喂 local_path | p33-p35 表明稀疏 anchor 会主导 estimator 并破坏推进。 |
| feed heading smoothing | p40-p42 表明 heading 中混有真实转弯和错误跳变，简单低通会阻断有效信息。 |
| builder 内 axis selection/hysteresis | p43/p45 表明感知层提前提交单一方向仍会导致 route 崩溃。 |
| 控制层磁跨线对齐 probe | p47 使 crossings/cycle 和 route 同时退化，保留负例，不进入主线。 |
| 只看 heading/横偏优化 | p44/p45 证明局部 heading 好不等于任务推进健康。 |

### 2.3 当前首要瓶颈

不是埋深估计器精度，也不是 phase 门限过严，而是：

```text
probe leg cycle 数量不少
  -> 真实磁跨线事件不足
  -> phase/cycle summary supply 不足
  -> 多假设 selector 只有部分窗口可用
  -> 控制层缺少可确认的连续目标
```

因此下一阶段不应继续扩大参数网格，而应先把候选、选择、确认、消费这四个 API 边界固定下来。

## 3. 目标数据流

### 3.1 分层契约


- shadow 通道可以记录所有候选。
- 低权重外推 feed 作为 p36 对照保留，但标注为 baseline experiment，不再扩展成默认机制。

- burial posterior 可以参与 hypothesis quality score，但先不改变控制。

### 2026-06-26：Parameterized forward zig-zag sweep

本轮实现参数化 shadow sweep，仍不接控制。实现方式是在 metrics 层用已记录的 `zigzag_probe_active`、`zigzag_probe_leg_flip_event`、`speed_mps` 与 `dt_s` 重建实际 probe leg 的行驶距离，然后扫描：

- crossing angle: `10° / 14° / 18° / 22°`
- leg duration multiplier: `1.0 / 1.5 / 2.0`

每组计算：

```text
route_delta = leg_distance * cos(angle) * multiplier
lateral_sweep = leg_distance * sin(angle) * multiplier
feasible = route_delta > 0.5m AND lateral_sweep >= 2.0m
```

1. 参数扫描证明“route-forward + 足够横扫”在几何上并不冲突。

   - route completion；

### 2026-06-26：ProbeBurstManager 显式状态机设计

用户选择方案 2：继续受控激进路线，但停止直接堆叠 p78 参数变体。当前实现把 probe burst 从 controller 内部的固定周期计时器抽出为 `ProbeBurstManager`，默认关闭，由独立状态机管理 burst / recovery 相位。

新增文件：

- `src/auv_mag_tracking/probe_burst_manager.py`
- `tests/test_probe_burst_manager.py`

状态定义：

| 状态 | 控制含义 | controller 动作 |
| --- | --- | --- |
| `IDLE_BASELINE` | 使用 p36 基线积累稳定 route-progress baseline | 不覆盖 p36 heading |
| `BURST_COLLECT_EVIDENCE` | 短时启用 decoupled lateral target，主动制造 forward magnetic evidence | `burst_active=True`，允许 decoupled lateral heading 覆盖 |
| `RECOVER_ROUTE` | 退出 probe 后拉回 route centerline，优先恢复任务推进 | `recovery_active=True`，使用 route-centerline recovery heading |
| `COOLDOWN` | 禁止连续 burst，等待 route 和感知状态重新稳定 | 不覆盖 p36 heading |

状态机输入：

- `time_s`：当前仿真时间。
- `route_progress_m`：车辆在名义 route 上的投影进度。
- `abs_cross_track_m`：相对当前 cable / route 参考的横偏绝对值；无有效横偏时输入 `inf`，禁止进入 burst。
- `evidence_available`：当前帧是否出现 progress-aligned candidate 或 dual-gate evidence。
- `enabled`：只在 `TRACK_ACTIVE`、非 reacquire、且 decoupled lateral 控制允许时为真。`LOCK_ALIGN` 阶段不允许 burst，避免控制覆盖破坏 track 建立。

转换门限（默认值，全部集中在 `ScenarioConfig.tracking`）：

| 门限 | 默认值 | 用途 |
| --- | ---: | --- |
| `probe_burst_manager_idle_min_duration_s` | `30.0` | baseline 至少稳定这么久才允许进入 burst |
| `probe_burst_manager_entry_min_route_delta_m` | `3.0` | idle 期间必须有正向 route progress |
| `probe_burst_manager_entry_max_abs_cross_track_m` | `80.0` | 宽 entry recovery 口径；默认 manager 仍关闭，不污染 p36 |
| `probe_burst_manager_burst_min_duration_s` | `4.0` | 即使出现 evidence，也至少采集这么久 |
| `probe_burst_manager_burst_max_duration_s` | `12.0` | 无 evidence 时的 burst 硬上限 |
| `probe_burst_manager_burst_target_evidence_count` | `1` | burst 退出到 recovery 所需 evidence 事件数 |
| `probe_burst_manager_recovery_min_duration_s` | `20.0` | recovery 至少持续时间 |
| `probe_burst_manager_recovery_target_route_delta_m` | `8.0` | recovery 内需恢复的 route progress |
| `probe_burst_manager_recovery_max_abs_cross_track_m` | `3.0` | recovery 完成时横偏约束 |
| `probe_burst_manager_recovery_timeout_s` | `120.0` | recovery 超时后进入 cooldown，避免卡死 |
| `probe_burst_manager_cooldown_duration_s` | `120.0` | 两次 burst 间隔 |

核心转换：

```text
IDLE_BASELINE
  -> BURST_COLLECT_EVIDENCE
     when idle_elapsed >= idle_min
      and route_delta >= entry_min_route_delta
      and abs_cross_track <= entry_max_abs_cross_track

BURST_COLLECT_EVIDENCE
  -> RECOVER_ROUTE
     when evidence_count >= target and burst_elapsed >= burst_min
      or burst_elapsed >= burst_max

RECOVER_ROUTE
  -> COOLDOWN
     when recovery_elapsed >= recovery_min
      and route_delta >= recovery_target_route_delta
      and abs_cross_track <= recovery_max_abs_cross_track
      or recovery_elapsed >= recovery_timeout

COOLDOWN
  -> IDLE_BASELINE
     when cooldown_elapsed >= cooldown_duration
```

controller 接入边界：

- `ProbeBurstManager` 只做策略相位决策，不生成 heading。
- `BURST_COLLECT_EVIDENCE` 复用已有 decoupled lateral target heading。
- `RECOVER_ROUTE` 复用已有 route-centerline recovery heading。
- `IDLE_BASELINE / COOLDOWN` 回落到 p36 基线。
- 状态机只在 `TRACK_ACTIVE` 中更新；`LOCK_ALIGN` 必须保持 p36 建立跟踪，不参与 burst。
- `enabled=False` 时无条件 reset 到 `IDLE_BASELINE`，并刷新当前时间和 route progress，防止进入 TRACK 时继承旧 baseline。

测试覆盖：

- idle 进入 burst 需要时间、route delta、横偏三重门限。
- burst 在最小时长 + evidence 后进入 recovery。
- burst 无 evidence 时按 timeout 进入 recovery。
- recovery 需要 route delta 和横偏恢复后进入 cooldown。
- disabled 会 reset；即使已经在 idle，也会刷新 baseline，避免重启后立即 burst。

实施边界：

- 本轮不新增 p78 参数变体，不把状态机当作新的网格搜索入口。
- 下一步若继续闭环评估，应先把 manager state / reason / elapsed / route_delta / evidence_count 进入 recorder，再做单个代表性 manager-enabled 实验。

- `ProbeBurstManager` 新增内部 `entry_abs_cross_track_m`，只在 `IDLE_BASELINE -> BURST_COLLECT_EVIDENCE` 时写入。
- `BURST_COLLECT_EVIDENCE / RECOVER_ROUTE / COOLDOWN` 期间，decision 返回冻结 entry XT。
- 回到 `IDLE_BASELINE` 时清空冻结值。
- 新增单测 `test_entry_cross_track_is_frozen_during_burst_and_recovery`，锁定该合同。

- `80m` 不是“勉强到 82.6%”的参数折中，而是能复现 `XT=inf` endpoint 行为的有界 entry/safe 门限。
- 下一步优先保留 `80m` 口径并做跨场景回归，而不是继续修改 recovery timeout 或放宽到 `100m`。

#### 2026-06-28：DR/INS 慢漂下的估计器门限与进度连续性保护

单独边界扫描显示，`case_maze_sparse_sonar_dr_ins_prior_distortion` 在 `mid` DR/INS 漂移、`prob_detection=0.20` 时原本能到 endpoint，但 route projection 在回折迷宫中发生跨 lane 跳变：

- 调优前：health `60.8`，route `99.5%`，endpoint yes，geometry no，`max_jump=236.9m`，large jumps `3`。
- 根因：prior correction 使用“观测点到当前先验全局最近点”的残差更新整条 route；在稀疏锚点和 DR/INS 慢漂下，回折 lane 的空间最近点不一定是拓扑正确点。

修正：

- 新增 `nominal_route_progress_guard_enabled` 及 lookback/lookahead 窗口，只在上一帧 route progress 附近投影。
- 新增 prior correction 门限：
  - `nominal_route_prior_correction_max_residual_m`
  - `nominal_route_prior_correction_max_step_m`
  - `nominal_route_prior_correction_max_heading_error_deg`
- sparse DR/INS 场景将单步平移修正限制为 `0.35m`，避免少数稀疏锚点一次性拖动整条先验。
- progress guard 只对 sparse / dropout DR/INS 场景开启；连续 sonar 场景保持全局投影，避免强观测被不必要的 progress 窗口限制。

- sparse sonar 在 `prob_detection >= 0.20` 时可承受 `mid` DR/INS 慢漂。
- `prob_detection=0.15` 仍低于当前可观测性边界；此时 endpoint 可达但仍有几何跳变，不能作为通过点。
- 后续若要支持 `prob_detection=0.15`，应增加锚点记忆或磁路径拓扑约束，而不是继续提高 correction gain。
